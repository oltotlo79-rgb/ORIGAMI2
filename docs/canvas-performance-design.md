# Canvas性能ベンチマーク設計

## 目的

要件`SIM-009`の10,000本以上の展開図について、生成件数だけではなく、RustからUIへの実データ転送、React側の表示準備、Canvas初回描画、継続再描画、選択判定を同じデータで検証する。

測定は開いているプロジェクト、revision、保存状態、Undo/Redo履歴を変更しない。ベンチマーク終了後は元の展開図へ戻る。

## 標準データ

- 要求辺数: 10,000
- 実辺数: 10,000
- 頂点数: 5,184
- 形状: 72 × 72頂点の格子から、行優先で水平辺、垂直辺の順に採用
- 線種: 行・列の偶奇による山折り／谷折り
- ID: `benchmark-v-N`、`benchmark-e-N`
- 上限: 100,000辺

Rustコマンドとブラウザ開発用fallbackは、ID、順序、座標、線種を同じ規則で生成する。応答は頂点と辺の実配列を含み、件数だけを返す疑似ベンチマークにはしない。

## UIで表示する測定値

| 値 | 定義 |
|---|---|
| ペイロード量 | 応答をUTF-8 JSONにしたときのバイト数 |
| 生成+転送 | コマンド要求直前から応答受領まで。Rust生成、serialize、IPCを含む |
| 変換 | 応答検証、頂点・辺の表示データ化、ペイロード量計算 |
| UI準備 | React更新からCanvas effect開始までの概算。空間索引と描画batchの構築を含む |
| 初描画 | Canvasのclear、紙、grid、全辺、全頂点を1回描く同期時間 |
| FPS | `requestAnimationFrame`で同じCanvasを30フレーム実再描画した実測値 |
| p95 | 30回の同期描画時間を昇順に並べた95 percentile |

FPS測定は明示的なrequest IDごとに1回だけ行う。通常編集で測定loopを動かさず、依存変更、request変更、unmount時には予約済みframeをcancelする。ブラウザ、GPU、画面解像度、device pixel ratioの影響を受けるため、値は測定環境と一緒に評価する。

## 描画方式

Canvas 2Dのstroke stateが同じ線を一つのpathへまとめる。通常線の描画層は背面から次の固定順とする。

1. 補助線
2. 山折り
3. 谷折り
4. 輪郭線
5. 切断線
6. 選択線

この固定層は10,000本を最大10回のstrokeで描くための仕様である。文書配列順をz-orderには使用しない。頂点、選択halo、gridもそれぞれpathをまとめる。

## 空間索引

静的で決定論的なBVHを用い、次を同じ索引から検索する。

- 頂点クリック
- 辺クリック
- 頂点スナップ
- 中点スナップ
- 辺スナップ

構築は現在の決定論sort方式で`O(n log² n)`、保持メモリは`O(n)`、疎な局所検索は平均`O(log n + k)`、最悪`O(n)`である。同距離は元配列順を既定権威とし、クリック半径は従来どおり境界を含まない。NaN／Infinityは候補にせず、巨大な有限線分は正規化した射影と安定な凸結合で扱う。

## 正しさと性能の回帰

- Rust／ブラウザの4辺fixtureと10,000辺fixture
- 応答の件数、ID一意性、端点参照、有限座標、非退化辺の検証
- 10,000点・10,000線の局所検索で候補検査数32未満
- indexed／unindexedスナップ30,000固定seed差分試験で結果不一致0
- threshold境界、同距離、accept、除外頂点、極小scale、巨大有限線分
- 10,000交互線を線種別batchへ集約する試験

実WebViewの数値は端末依存のためunit testの固定時間合否には使用しない。Windows／macOSの配布候補で測定値を収集し、基準PCと正式な応答時間を別途確定する。

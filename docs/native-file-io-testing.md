# ネイティブ `.ori2` ファイル操作の試験境界

## 目的

ネイティブファイルダイアログを自動操作しなくても、`.ori2` の読込み・保存とプロジェクト状態の更新をWindows/macOS CIで回帰試験できるようにする。

## 責務の分離

ファイル操作を次の境界に分ける。

1. Tauriダイアログはローカルパスの選択とキャンセルだけを担当する。
2. `load_project_file` はファイルサイズ制限、読込み、コンテナ検証を行う。この段階ではプロジェクト状態を変更しない。
3. `apply_loaded_project_file` はダイアログ開始時のproject IDとrevisionを再照合してから、検証済みドキュメントを適用する。
4. `save_project_to_path` はデータ生成、ステージ書込み、同期、再検証、原子的置換がすべて成功した後にだけ、保存先・保存revision・保存済みドキュメントを更新する。
5. `save_project_as_selected_path` はproject IDとrevisionを再照合してから拡張子を正規化し、同じ保存処理を使用する。

WebViewへ任意パスを受け取る試験専用IPCは追加しない。製品版のファイルシステム権限境界は従来どおりRust側に限定する。

原子的置換は、macOSなどの非Windows環境では`atomic-write-file`を使用する。Windowsでは同ライブラリの汎用実装が既存ファイルを置換できないため、保存先と同じディレクトリに排他的なステージファイルを作る。外部プロセスの書込み・削除・rename共有を禁止した同じハンドルで、書込み、同期、ディスク上の内容の再読込み、`.ori2`再検証を行う。検証したハンドル自体を`SetFileInformationByHandle(FileRenameInfo)`で保存先へ置換するため、検証後にステージのパスだけを別ファイルへ差し替える競合も起きない。通常の失敗時はハンドルを先に閉じてステージファイルをDropで除去し、既存ターゲットを変更しない。プロセスの強制終了や外部プロセスによる権限変更では一時ファイルが残る可能性があるため、起動時清掃は復旧機能と合わせて追加する。

## 自動試験マトリクス

すべてOSの一時ディレクトリ内で、本番と同じ読書き関数を使用する。

| シナリオ | 自動検証内容 |
|---|---|
| 開く | 保存済み`.ori2`を読込み、identity、紙設定、保存先、clean状態を復元し、旧プロジェクトの履歴を引き継がない |
| 別名保存 | `.ori2`拡張子の正規化、再読込み可能性、保存情報の更新、既存Undo/Redo履歴の維持 |
| 通常保存・上書き | 既存ファイルを有効な`.ori2`で置換し、再編集後の再上書き、内容一致、履歴維持を確認 |
| Windowsステージ保護 | 保存中のステージファイルへの別writerとrenameをsharing violationで拒否し、検証済みハンドル自体を置換対象にし、ハンドル解放後にステージを清掃する |
| 破損入力 | コンテナ検証で拒否し、現在のドキュメント、パス、dirty状態、revision、Undo/Redo履歴を変更しない |
| 開く途中の編集 | ダイアログ開始時よりrevisionが進んだ場合、読込み済みデータを適用せず最新状態と履歴を維持する |
| 保存失敗 | ファイルへ置換できない保存先で原子保存を失敗させ、現在状態と既存ターゲットを維持する |
| 別名保存途中の編集 | 古いrevisionの保存をファイル作成前に拒否し、現在状態と選択先を変更しない |

既存CIのRustジョブはWindowsとmacOSの両方で `cargo test --workspace --locked --all-targets` を実行するため、追加した試験も両OSで自動実行される。

## 実機で残す確認

次はOSネイティブUIおよび権限との統合境界なので実機確認を残す。

- 開く・保存・別名保存ダイアログの表示、初期フォルダー、フィルター、推奨ファイル名
- ダイアログのキャンセル操作と、キャンセル後に編集状態が維持されること
- OSの上書き確認UI、読み書き権限エラー、ネットワークドライブや外部ボリューム固有の表示
- Windows/macOSの署名済み配布ビルドにおけるダイアログUX

ダイアログキャンセル後の応答生成自体は状態を変更しない純粋な既存経路だが、「キャンセルボタンを押してその経路へ到達すること」は実機境界として扱う。
